输入：视觉状态 ball, robots_our, robots_opp, referee_state, time_left

      计算所有 pass 点的 S_pass
      计算当前 shoot 的 S_shoot 和 canShoot
      若 canShoot 且 S_shoot > S_pass:
          执行 shoot
      否则:
          执行 passball + receiveball

      执行 getball

      计算 intercept / block / press 三类得分

6. 其余机器人执行 defend 或 off-ball running
7. 用角色分配模型重新匹配机器人和任务
8. 将任务目标交给技能层和路径规划层执行

# 16. 结论

这些任务都有可以落地的数学模型。getball 主要是前置点和到达时间最小化；passball 主要是球速反推、传球点搜索和拦截时间约束；receiveball 是球-机器人到达时间匹配；getballopp 可以建模为对手控球概率、截球点、封堵线和压迫点选择；shoot 是有效射门角与拦截预测的融合；defend 和 goalie 是威胁评分与射门线封堵；无球跑位是区域价值函数最大化；角色分配可用匈牙利算法求总代价最小。

比赛中真正有效的做法，是把这些模型拆成若干可调权重的评分函数，然后用比赛日志不断拟合参数。这样既能满足省赛定位球“15秒内传接后射门”的稳定性要求，也能扩展到国赛3V3中的抢球、防守和攻防转换。

资料范围：本文只使用网上公开资料，包括比赛规则文件、赛事报道、SOM/iLoboke相关文章、RoboCup Small Size League 公开资料、Rocos文档以及公开论文。报告中对 getballopp 的解释属于基于任务命名和比赛语境的工程推断，未把它写成官方公开定义。

# 摘要

iLoboke足球机器人比赛本质上是一个小型多智能体足球系统：机器人依赖上方视觉系统获得球、己方机器人和对方机器人的位置；策略端依据比赛状态运行脚本；技能和规划层把“拿球、传球、射门、接球、拦截、防守”等任务转换成速度、朝向、踢球和吸球等底层动作。第二十七届中国机器人及人工智能大赛规则明确指出，创非凡iLoboke足球项目重点考察图像处理、自动决策、路径规划、自主避障、C++/Lua混合程序设计，以及多智能体协同和动态环境控制能力1。

从比赛任务看，省赛常见目标是两个进攻机器人在五个固定点位完成定位球战术，除点球外射门前至少传接一次，且要求在15秒内完成1。国赛则升级到3V3对抗，要求在原有配合基础上增加对抗、防守等策略1。因此，先进算法不应只理解为某个单独函数，而应理解为“感知、状态估计、任务切换、角色分配、路径规划、传射决策、防守压迫、实时调参”的组合系统。

# 一、比赛平台与系统基础

# 1.1 iLoboke与SOM平台

iLoboke挑战赛规则文件说明，参赛学校使用 iLoboke足球机器人专用软件 SOM3.4.2 编写程序并在现场运行，平台提供虚拟仿真环境，因此无实物也可进行调试1。较早的iLoboke挑战赛方案也说明，SOM软件是与北京启创远景科技有限公司iLoboke足球机器人配套的操作管理软件，为用户提供人机交互和二次开发平台2。

在软件结构上，SOM后端主要实现决策调度，并分为策略脚本层、技能规划层和基础SDK层；其中策略脚本层使用 Lua，技能规划层和基础SDK层使用 C++2。这说明 getball、passball 一类任务通常不是孤立动作，而是 Lua 层组织比赛状态、C++ 层执行运动和踢球技能的协作结果。

# 1.2 典型硬件与赛场约束

第二十七届规则规定，iLoboke赛场有效尺寸为6m×4m，场地上方约3.2m处架设高速相机采集图像，比赛用球为橘黄色高尔夫球1。省赛中通常由2台进攻机器人完成5个固定点位的定位球进攻，防守侧包含守门员和场景放置的防守机器人；决赛阶段则为双方各3台机器人对抗1。

这些规则直接决定算法目标：省赛强调稳定、快速、少犯规的定位球战术；国赛强调对抗条件下的持球、传球、拦截、压迫和防守转换。赛事报道也显示，优秀队伍会围绕“传球速度优化”“精准识别位置”“高速传递”等方面持续调试运动算法3。

# 二、整体算法架构

小型足球机器人系统通常可拆为四个闭环：视觉感知、策略决策、通信下发、机器人执行。Rocos文档对小型足球机器人系统的描述与iLoboke场景高度相似：外部全局图像系统完成识别，核心决策系统部署策略与规划，通信模块下发指令，机器人硬件执行运动、踢球、吸球盘带等动作4。RoboCup SSL官方技术概览也说明，SSL-Vision负责把场上相机图像转换为机器人和球的坐标，再通过网络广播给队伍；同时原始视觉检测会有噪声、丢失和无速度信息，因此还需要进一步处理或跟踪5。

# 四、getball：从“冲向球”到“稳定控球”

# 4.1 任务定义

getball 的目标是让机器人从当前位姿到达合适的抓球位姿，并在不犯规、不把球顶飞、不被对手抢先触球的情况下获得控球权。基础写法可能只是让机器人朝球移动，但比赛中的先进 getball 应至少解决四个问题：从哪一侧接近球、以什么朝向接近球、何时启动吸球/盘带、失败后如何恢复。

# 4.2 抓球前置点算法

简单追球容易出现“从侧面或后方顶球”的问题。更稳的做法是根据后续动作确定目标方向，例如传球给队友、射门到球门或避开防守者，然后在球的反方向生成一个前置点：

目标方向 dir = normalize(target - ball)
抓球前置点 pre = ball - d_pre * dir
机器人目标朝向 theta = angle(target - robot)

其中 target 可以是接球队友、球门空档或下一状态需要的目标点。机器人先到 pre，再沿 dir 方向小速度贴近球。这样做的好处是，机器人头部、球和后续传射目标大致共线，拿到球后可以直接进入 passball 或 shoot。

# 4.3 状态切换与容错

Rocos文档说明，Play中的 switch 函数每帧执行，可依据距离、角度、速度、位置、是否持球、是否踢球、球出界、球消失、机器人数量变化、裁判盒信息和时间等条件切换状态9。在 getball 中，先进状态机不应只判断“离球距离小于阈值”，而应综合判断：

  1. 球是否位于机器人前方可吸球扇区。
  2. 机器人朝向与球-目标方向夹角是否足够小。
  3. 红外或控球状态是否表明球已被吸住。
  4. 机器人和球是否接近边界或禁区。
  5. 对手是否会更早到达球。

推荐状态机如下：

    [*] --> ApproachBehindBall
    ApproachBehindBall --> AlignToTarget: 到达抓球前置点
    AlignToTarget --> SlowCapture: 朝向误差小
    SlowCapture --> Controlled: 控球成功

# 4.4 关键改进点

第一，getball 要和路径规划联动。禁区、边界、对方机器人和己方队友都应进入代价地图，避免为了拿球而进入禁区或碰撞防守机器人。第二，getball 要具备低速末端控制，接近球时降低速度并锁定朝向，减少把球推偏的风险。第三，getball 应对视觉噪声鲁棒：SSL官方资料指出视觉检测有噪声和丢失问题5，因此工程上应加入位置滤波、球速估计、短时丢球保持和异常重规划。

# 五、passball：从“把球踢给队友”到“动态传射决策”

# 5.1 规则驱动的传球价值

省赛定位球规则要求，除点球外，射门前至少完成一次传接配合1。这意味着 passball 不是可选动作，而是得分链路的一部分。公开赛事报道中也提到，参赛队会围绕传球速度优化、位置识别和高速传递进行调试3。

# 5.2 传球力度与球模型

ZJUNlict在RoboCup SSL 2019冠军论文中指出，传球力度过小会给对手足够时间拦截，力度过大则可能导致队友无法在有限时间内接球，因此需要基于球模型计算合适传球力度10。论文把传球时间拆为接球机器人到达时间和到达后的调整缓冲时间，并结合传球距离估计初速度10。

映射到iLoboke比赛中，passball 可以采用如下计算框架：

候选接球点 p
传球距离 d = |p - passer|
接球者到达时间 t_robot = time_to_reach(receiver, p)
缓冲时间 t_buffer = 0.2s ~ 0.4s
目标传球时间 t_pass = t_robot + t_buffer
传球速度 v = ball_model_inverse(d, t_pass)

在省赛中，两个进攻机器人通常预设“拿球者-接球者-射门者”的链路。先进做法不是固定传球点，而是在每个固定点位启动后，根据球、守门员、防守机器人和己方接球者的位置实时选择接球点。

# 5.3 传球点搜索与拦截风险

ZJUNlict论文提出的动态传射算法包括四个模块：传球点模块计算所有传球点可行性并选择最佳点，跑位点模块寻找进攻威胁更高的位置，决策模块判断何时传球或射门，技能模块保证动作执行准确10。该论文还以拦截时间、接球点射门角度、接球点到球门距离、射门折射角、己方先到与对方先到的时间差等特征评价传球策略10。

因此，passball 的先进评分函数可写为：

score(p) =
  w1 * pass_safety(p)
+ w2 * receiver_arrival_margin(p)
+ w3 * shooting_angle_after_receive(p)
+ w4 * distance_to_goal_value(p)
+ w5 * no_foul_margin(p)
- w6 * collision_risk(p)

其中 pass_safety 由“对手是否能在球到达前截到球”决定，receiver_arrival_margin 表示己方接球者比对手更早到达的时间差，no_foul_margin 用于避免进攻机器人进入禁区、二次触球或碰撞防守机器人等犯规。

# 5.4 接球同步

passball 不只是传球者动作，还包括接球者的 receiveball。Rocos的任务解析文档显示，一个Task可以返回执行函数和匹配位置，Lua层还可配置踢球模式、目标朝向、踢球精度、力度和特殊flag，从而把非运动指令与运动本身解耦11。这启发我们把传球设计为“双机器人同步任务”：

  1. 传球者根据接球者预计到达时间计算球速。
  2. 接球者提前移动到接球点，并朝向来球方向。
  3. 接球者在球到达前进入低速等待或微调状态。
  4. 传球者踢球后，接球者切换到吸球或一脚射门状态。

# 六、getballopp：对手控球场景下的抢球、封堵与压迫

# 6.1 资料边界

本次在线检索没有找到 getballopp 的官方公开定义。按常见命名习惯，它很可能表示 “get ball opponent” 或“对方控球/对方取球时的处理任务”。在报告中，getballopp 被定义为：当对手更接近球、对手准备接球、或对手即将形成射门/传球威胁时，己方机器人执行抢球、截球、封堵传球线或压迫持球人的综合任务。

# 6.2 对手触球时间预测

先进 getballopp 的第一步是判断“谁会先碰到球”。ZJUNlict论文中把控球率计算与双方拦截时间联系起来：若一方拦截时间更短，则球被认为由该方控制10。这给 getballopp 一个清晰指标：

t_our = min time_to_intercept(己方机器人, ball_trajectory)
t_opp = min time_to_intercept(对方机器人, ball_trajectory)
if t_opp + margin < t_our:
    进入 getballopp / 压迫防守
else:
    进入 getball / 主动拿球

若对手先到，直接冲球往往会造成碰撞或被轻松过掉，更好的动作是抢占对手下一步传球/射门路径。

# 6.3 三类策略

第一类：抢截球。 当球是滚动球或无人稳定控球时，己方机器人计算球轨迹与自身可达区域的交点，选择最早可达且不越界、不入禁区的拦截点。

第二类：封传球线。 当对手持球但射门角度不大时，机器人站到“对手持球点-潜在接球者”或“对手持球点-球门”的连线上，优先阻断高威胁传球路线。

第三类：压迫持球人。 当对手拿球且转身空间有限时，机器人从侧前方逼近，迫使对手向低威胁方向移动。这里不能直接撞击对方机器人，因为第二十七届规则把进攻机器人碰撞防守机器人列为犯规之一1；在3V3对抗中也需要用规则安全的方式防守。

# 6.4 getballopp 状态机

stateDiagram-v2
    [*] --> EstimatePossession
    EstimatePossession --> InterceptBall: 球未被稳定控制
    EstimatePossession --> BlockLane: 对手准备传球
    EstimatePossession --> PressHolder: 对手稳定持球
    InterceptBall --> ClearOrPass: 己方截到球
    BlockLane --> GetBall: 对手传球被破坏
    PressHolder --> RetreatDefend: 对手突破或己方失位

getballopp 的先进性不在于“更快冲向对手”，而在于用预测和几何控制缩小对手选择空间。它需要和 defend、goalie、passball 共享同一套威胁评分：对手离球门越近、射门角越大、传球线越通畅，则防守动作越激进；反之则保持阵型，避免无谓出击。

# 七、射门、定位球与无球跑位算法

# 7.1 射门决策

射门不是越早越好。ZJUNlict论文指出，随意射门可能导致丢失球权，条件过严又会降低进攻效率，因此需要结合射门角和拦截预测来判断是否射门10。iLoboke省赛中，射门前还要满足传接配合要求，且直接射门会被判犯规或进球无效1。

推荐射门判断：

can_shoot =
    has_required_pass
and shot_angle > angle_threshold
and opponent_intercept_time(ball_to_goal) > ball_arrival_time + margin
and robot_not_in_penalty_area
and shot_power_within_limit

# 7.2 定位球战术

省赛的五个点位包含后场区、开球点、前场区、角球区、点球点，具体点位赛前不公布1。这要求算法不能只写死坐标，而应根据开赛后裁判放球点实时生成战术。一个稳健的省赛流程是：

  1. 读取球点和障碍机器人位置。
  2. 由最近且角度最优的机器人执行 getball。
  3. 另一台机器人计算接球点，避开防守机器人和禁区。
  4. 持球者执行 passball。
  5. 接球者判断直接射门或微调后射门。
  6. 全程监控15秒倒计时，若超过安全时间则选择快速射门或弃守当前战术。

# 7.3 无球跑位

ZJUNlict论文把前场划分为动态区域，并根据球的位置改变区域；然后把区域栅格化，对每个点按接近球门、离球距离、面对球门角度、对手防守到达时间、避免陷入对手防守等标准评分10。在iLoboke中，这类算法可用于 GoRecePos、ReceiveBall 或自定义跑位任务：接球者不是站在固定点，而是实时寻找“更容易接球、更容易射门、更不容易被拦截”的点。

# 7.4 反盯防移动

ZJUNlict还提出 Drag skill：当机器人到达目的地后容易被对手盯防时，通过先诱导防守者加速，再反向加速拉开距离，以提升安全接球概率10。在iLoboke 3V3对抗中，类似思路可用于接球机器人：不要在接球点静止等待，而是在不越位、不入禁区、不撞人的前提下做小范围横移，使对手难以提前卡死传球线。

# 八、角色分配与多机器人协同

角色分配决定哪个机器人去拿球、哪个机器人接球、哪个机器人补防。Rocos文档说明，在多智能体场景中，匹配规则用于把角色任务分配给具体机器人；实时匹配可保证总代价最小，底层使用二分图匹配中的匈牙利算法12。这类机制对iLoboke也很重要，因为比赛中机器人可能因位置、电量、碰撞或识别异常导致最优执行者变化。

一个高级角色分配代价函数可以包括：

cost(robot, role) =
  a1 * 到任务目标距离
+ a2 * 当前朝向误差
+ a3 * 到达时间
+ a4 * 可能碰撞风险
+ a5 * 是否带球/是否刚踢球
+ a6 * 禁区和边界惩罚
+ a7 * 通信或硬件状态惩罚

在省赛中，角色可较稳定：一台主抓球/传球，一台主接球/射门，守门员由组委会或己方固定。国赛3V3中，角色应动态切换：前锋丢球后可立刻转为压迫者，中场可转为接应点，守门员在球远离危险区时可上提参与出球。

# 九、路径规划与避障

路径规划层把“去球后方”“去接球点”“封堵传球线”等目标转换为可执行轨迹。SOM规则要求场地有禁区、边界、球门和防守机器人，且进攻机器人进入禁区、二次触球、碰撞防守机器人都会影响成绩1。因此，路径规划不仅要快，还要规则安全。

推荐采用分层规划：

  1. 全局目标生成：由 getball、passball、getballopp 计算目标点。
  2. 规则过滤：删除禁区内、边界外、可能碰撞的目标点。
  3. 局部避障：对动态机器人设置膨胀半径，避免高速贴近。
  4. 速度规划：接近球、接近对手、接近边界时降速。
  5. 重规划触发：球速变化、视觉丢失、对手突然靠近、队友位置变化时重算路径。

对比赛成绩而言，路径规划的评价指标不是单纯最短路径，而是“最短时间内完成有效传射且不犯规”。这也是为什么优秀算法常把路径规划和传球决策合并考虑：某个传球点看似角度好，但如果接球者到达时间太长或路径穿过对手防守区，就不是好点。

# 十、针对不同赛段的策略建议

# 10.1 省赛定位球

省赛目标清晰：五个点位、15秒、至少一次传接配合、进球优先、用时次之1。建议采用“少状态、高确定性”的策略：

场景 首选策略 重点算法
后场区 稳定传球后远射或推进射门 长距离传球力度、接球者提前跑位
开球点 快速一传一射 固定开球模板、射门角选择
前场区 短传后低延迟射门 接球同步、守门员角度判断
角球区 斜传到门前空档 传球线避障、接球者横向跑位
点球点 直接射门或假动作射门 射门角、守门员位置预测

# 10.2 国赛3V3对抗

中国大学生在线报道提到，相关赛事决赛阶段需要自行携带机器人，并且国赛赛制升级为3V3对抗，每队派出3台机器人进行约5分钟比赛，并新增防守策略设计13。规则文件也说明，国赛3V3中双方共有六台机器人，参赛队需在原有配合基础上增加对抗和防守策略1。

3V3中建议使用“攻防状态机”：

flowchart TD
    A["开球 / 普通比赛"] --> B{"己方是否先到球"}
    B -->|是| C["getball 控球"]
    C --> D{"传球窗口是否安全"}
    D -->|是| E["passball + receiveball"]
    D -->|否| F{"射门窗口是否安全"}
    F -->|是| G["shoot"]
    F -->|否| H["off-ball running / 重新组织"]
    B -->|否| I["getballopp 压迫或截球"]
    I --> J{"对手威胁等级"}
    J -->|高| K["block lane / goalie 协防"]
    J -->|低| L["保持阵型 / 等待失误"]

# 十一、主要风险与改进方向

风险 现象 改进方向
视觉误差 球坐标跳动、机器人误判持球 滤波、短时预测、状态滞回
抓球不稳 机器人绕球、顶球、错过球 前置点、低速末端、朝向闭环
传球失败 球速过大或过小、队友没到位 球模型、到达时间预测、传球缓冲
被对手拦截 传球线被封堵 拦截时间评估、传球点搜索
犯规 入禁区、二次触球、碰撞 规则约束加入规划代价
状态切换抖动 反复在getball/passball之间切换 状态滞回、最短状态保持时间
国赛对抗不适应 只会定位球，不会防守 getballopp、盯防、守门、阵型切换

未来可以重点改进三方面。第一，用在线日志自动调参：记录每次传球距离、力度、接球成败、射门角度和用时，回放后优化权重。第二，引入对手建模：统计对手常用传球方向、守门员移动习惯、压迫强度，让 getballopp 更像“预判”而不是“追赶”。第三，提高技能层鲁棒性：针对抓球、接球、踢球分别做小规模测试集，使任务在仿真和实物之间表现更一致。

# 结论

iLoboke足球机器人比赛的先进算法不是单个函数的堆叠，而是围绕比赛目标形成的闭环系统。getball 的核心是稳定获得球权，关键在于前置点、朝向控制、吸球判断和异常恢复；passball 的核心是满足规则并创造高质量射门机会，关键在于球模型、传球点搜索、接球同步和拦截风险评估；getballopp 虽未找到官方公开定义，但从3V3对抗需求看，它应被设计为对手控球时的抢截、封堵和压迫任务，核心在于预测双方拦截时间、控制传球线和避免犯规。

如果要在比赛中取得稳定成绩，建议把算法优先级排为：先保证 getball 和 passball 的成功率，再优化射门决策和定位球模板，最后加入 getballopp、防守阵型和无球跑位。省赛靠稳定传射拿分，国赛靠攻防切换和对抗决策拉开差距。

# AI 专业名词解释

这份文档用于快速理解 AI、LLM、API、Agent、MCP、Skill 等相关概念。术语按主题分组，适合初学者、产品经理、开发者和需要与 AI 系统协作的人阅读。

# 1. 基础概念

# AI

AI 是 Artificial Intelligence 的缩写，中文通常称为“人工智能”。它泛指让机器具备感知、理解、推理、生成、决策或执行任务能力的技术体系。

常见应用包括：

  • 文本生成
  • 图像生成
  • 语音识别
  • 智能客服
  • 推荐系统
  • 自动驾驶
  • 数据分析
  • 代码生成

# Machine Learning

Machine Learning，简称 ML，中文是“机器学习”。它是 AI 的一个重要分支，核心思想是让模型从数据中学习规律，而不是完全依赖人工编写规则。

例如：给模型大量图片和标签，模型可以学习识别猫、狗、汽车等对象。

# Deep Learning

Deep Learning，中文是“深度学习”。它是机器学习的一类方法，通常使用多层神经网络来处理复杂数据。

深度学习常用于：

  • 图像识别
  • 语音识别
  • 自然语言处理
  • 大模型训练
  • 自动驾驶感知

# Model

Model，中文是“模型”。在 AI 中，模型可以理解为经过训练后具备某种能力的程序。

例如：

  • 文本模型可以回答问题、写文章、写代码
  • 图像模型可以生成图片
  • 语音模型可以识别或合成声音
  • 多模态模型可以同时理解文本、图片、音频、视频

# LLM

LLM 是 Large Language Model 的缩写，中文是“大语言模型”。它是一类能够理解和生成自然语言的 AI 模型。

常见能力包括：

  • 对话
  • 总结
  • 翻译
  • 写作
  • 代码生成
  • 信息抽取
  • 推理
  • 工具调用

# Multimodal Model

Multimodal Model，中文是“多模态模型”。它可以处理多种类型的信息，例如文本、图片、音频、视频、表格等。

例如：用户上传一张截图并提问，模型能够读取图片内容并给出解释。

# Generative AI

Generative AI，中文是“生成式 AI”。它指能够生成新内容的 AI，例如文字、图片、音乐、视频、代码等。

典型例子：

  • 生成文章
  • 生成海报
  • 生成 PPT
  • 生成程序代码
  • 生成短视频脚本

# 2. Token 与上下文

# Token

Token 是 AI 模型处理文本时的基本单位。它不一定等于一个字、一个词或一个字符，而是模型分词后得到的小片段。

例如：

  • 英文中，一个单词可能是一个 token，也可能被拆成多个 token
  • 中文中，一个汉字、词语或标点都可能成为 token
  • 空格、换行、标点也可能被计算为 token

Token 影响：

  • 输入长度
  • 输出长度
  • 费用计算
  • 模型能记住的上下文范围
  • 接口调用速度

# Tokenization

Tokenization，中文可称为“分词”或“标记化”。它是把文本转换为 token 的过程。

例如一句话：

I love AI.

模型不会直接按人类阅读方式理解，而是先转换成一组 token，再进行计算。

# Input Token

Input Token 是用户输入给模型的 token，包括问题、上下文、系统指令、历史对话、工具返回内容等。

输入越长，消耗的 token 越多，成本通常也越高。

# Output Token

Output Token 是模型生成回答时产生的 token。

例如模型写了一篇 1000 字文章，这些生成内容就会被计算为输出 token。

# Context Window

Context Window，中文常称为“上下文窗口”。它表示模型一次调用中最多能处理多少 token。

上下文窗口越大，模型能同时参考的信息越多。

例如：

  • 短上下文模型适合简单问答
  • 长上下文模型适合阅读长文档、分析代码库、处理合同、总结会议记录

# Max Tokens

Max Tokens 通常表示一次调用中允许模型最多生成多少 token。

如果设置过小，模型回答可能被截断；如果设置过大，成本和响应时间可能增加。

# 3. Prompt 与消息角色

# Prompt

Prompt，中文常称为“提示词”或“提示”。它是用户给 AI 的输入指令，用来告诉模型要完成什么任务。

好的 prompt 通常包含：

  • 目标
  • 背景
  • 输入材料
  • 输出格式
  • 约束条件
  • 示例

# System Prompt

System Prompt 是系统级指令，通常用于定义模型的身份、行为边界、回答风格和优先规则。

例如：

你是一名严谨的法律助理，请用简洁、准确的语言回答。

# User Message

User Message 是用户发给模型的内容，也就是用户问题或任务说明。

# Assistant Message

Assistant Message 是模型生成的回复内容。

# Developer Message

Developer Message 是开发者给模型的中高优先级指令，通常用于控制应用中的模型行为，例如输出格式、安全规则、工具使用方式等。

# Prompt Engineering

Prompt Engineering，中文是“提示词工程”。它是设计、优化和测试 prompt 的方法，使模型更稳定、更准确地完成任务。

常见技巧包括：

  • 明确角色
  • 指定输出格式
  • 提供示例
  • 拆分复杂任务
  • 说明边界条件
  • 要求模型先分析再输出

# Few-shot Prompting

Few-shot Prompting 是在 prompt 中提供少量示例，让模型模仿示例完成类似任务。

# Zero-shot Prompting

Zero-shot Prompting 是不给示例，直接让模型完成任务。

# 4. API 与开发接入

# API

API 是 Application Programming Interface 的缩写，中文是“应用程序编程接口”。它允许一个程序调用另一个服务的能力。

在 AI 场景中，API 通常用于：

  • 调用大模型生成文本
  • 调用图像生成模型
  • 调用语音识别服务
  • 上传文件让模型分析
  • 让模型调用工具或函数

# API Key

API Key 是访问 API 服务的密钥。它相当于一把“访问凭证”，用于证明调用者有权限使用某个服务。

注意事项：

  • 不要把 API Key 写进公开代码仓库
  • 不要截图或公开分享 API Key
  • 生产环境应使用环境变量或密钥管理服务保存
  • 如果泄露，应立即删除或轮换

# Key

Key 在不同语境中含义不同。在 AI 开发中，通常指 API Key、访问密钥、加密密钥或配置项中的键名。

常见含义：

  • API Key：访问接口的凭证
  • Secret Key：敏感密钥
  • Public Key：公钥
  • Private Key：私钥
  • Config Key：配置字段名

# Endpoint

Endpoint，中文可称为“接口地址”或“服务端点”。它是 API 请求发送到的 URL。

例如：

https://api.example.com/v1/chat/completions

# Request

Request 是“请求”。当程序调用 AI API 时，会向服务端发送 request，其中通常包含模型名称、用户输入、参数、文件或工具配置。

# Response

Response 是“响应”。它是 API 返回的结果，通常包含模型生成内容、状态信息、错误信息、token 用量等。

# SDK

SDK 是 Software Development Kit 的缩写，中文是“软件开发工具包”。它封装了底层 API 调用，让开发者更方便地使用某个平台。

例如使用 SDK 时，开发者可能只需要写几行代码就能调用模型。

# Rate Limit

Rate Limit 是“速率限制”。它限制一定时间内最多可以调用多少次 API，或者最多可以消耗多少 token。

常见限制包括：

  • 每分钟请求数
  • 每分钟 token 数
  • 每天调用额度
  • 并发请求数量

# Quota

Quota 是“配额”。它表示账户、项目或服务在某个周期内可使用的资源上限。

# Billing

Billing 是“计费”。AI 服务通常按照输入 token、输出 token、图片数量、音频时长、模型类型或工具使用量计费。

# 5. 模型调用参数

# Temperature

Temperature 是控制模型输出随机性的参数。

一般理解：

  • 数值低：回答更稳定、更保守
  • 数值高：回答更多样、更有创造性

常见用法：

  • 事实问答、代码、数据分析：使用较低 temperature
  • 创意写作、头脑风暴：可以使用较高 temperature

# Top P

Top P 是另一种控制输出随机性的参数，也叫 nucleus sampling。

它控制模型在生成下一个 token 时，从概率累计达到某个阈值的候选集合中选择。

# Stop Sequence

Stop Sequence 是“停止序列”。当模型生成内容遇到指定字符串时，会停止继续生成。

# Seed

Seed 是随机种子。某些模型支持设置 seed，使输出更容易复现。

注意：即使设置 seed，模型服务更新或参数变化后，也不一定完全一致。

# Streaming

Streaming 是“流式输出”。模型不是等完整回答生成完才返回，而是一边生成一边返回。

适合：

  • 聊天应用
  • 长文本生成
  • 需要更快首字响应的产品

# 6. Agent、Tool 与函数调用

# Agent

Agent，中文常称为“智能体”。它不仅能回答问题，还能根据目标规划步骤、调用工具、读取文件、执行操作，并根据结果继续调整。

简单理解：

  • 普通聊天模型：主要负责回答
  • Agent：可以围绕目标持续行动

例如一个代码 Agent 可以：

  • 阅读项目文件
  • 修改代码
  • 运行测试
  • 分析错误
  • 继续修复

# Tool

Tool 是“工具”。它是模型可以调用的外部能力。

常见工具包括：

  • 搜索网页
  • 读取文件
  • 写入文件
  • 查询数据库
  • 调用 API
  • 执行代码
  • 生成图片

# Function Calling

Function Calling，中文常称为“函数调用”。它允许模型按照开发者定义的函数结构输出参数，由程序执行对应函数。

例如用户说：

帮我查一下今天上海天气。

模型可以生成类似：

{
  "function": "get_weather",
  "arguments": {
    "city": "Shanghai",
    "date": "today"
  }
}

然后应用程序调用天气接口并把结果返回给模型。

# Tool Calling

Tool Calling 是更宽泛的“工具调用”。它不只包括函数，还可以包括搜索、文件系统、数据库、浏览器、代码执行器等工具。

# Planner

Planner 是“规划器”。在 Agent 系统中，它负责把复杂目标拆解成多个步骤。

例如：

目标：分析竞品网站
步骤：
1. 收集竞品列表
2. 抓取官网信息
3. 比较功能、价格、定位
4. 总结机会点

# Executor

Executor 是“执行器”。它负责执行规划器生成的步骤，例如调用搜索工具、运行代码、写文件等。

# Memory

Memory 是“记忆”。它用于让 AI 系统在多次对话或长期使用中保留信息。

常见类型：

  • 短期记忆：当前对话上下文
  • 长期记忆：用户偏好、项目背景、历史任务
  • 外部记忆：数据库、向量库、文件系统

# 7. Skill 与插件

# Skill

Skill，中文可理解为“技能”。在 AI Agent 系统中，Skill 通常是一组专门的操作说明、流程、脚本或资源，用来让 AI 更擅长完成某类任务。

例如：

  • 数据分析 Skill
  • 写作 Skill
  • 做 PPT Skill
  • 安全审计 Skill
  • Excel 处理 Skill
  • 网页搜索 Skill

一个 Skill 可能包含：

  • 使用规则
  • 工作流程
  • 示例
  • 模板
  • 脚本
  • 参考资料

# Plugin

Plugin，中文是“插件”。插件通常是一组可安装的扩展能力，可以为 AI 应用增加工具、技能、界面或外部服务连接能力。

Skill 更偏“让 AI 学会一种工作方式”，Plugin 更偏“给系统安装一组能力”。

# Workflow

Workflow，中文是“工作流”。它表示完成某个任务的一套固定步骤。

例如“会议纪要工作流”可能包括：

  • 读取会议记录
  • 提取议题
  • 总结决策
  • 提取行动项
  • 标注负责人和截止日期

# Automation

Automation，中文是“自动化”。它指让系统在特定时间、条件或事件触发时自动执行任务。

例如：

  • 每周生成周报
  • 每天早上汇总待办
  • 每小时检查网站状态
  • 定期扫描安全风险

# 8. MCP 相关概念

# MCP

MCP 是 Model Context Protocol 的缩写，中文可称为“模型上下文协议”。它是一种让 AI 应用与外部工具、数据源、服务进行标准化连接的协议。

简单理解：MCP 像是 AI 应用和外部世界之间的统一连接标准。

通过 MCP，模型可以更容易地访问：

  • 文件
  • 数据库
  • 代码仓库
  • 浏览器
  • 企业系统
  • 搜索服务
  • 本地工具

# MCP Server

MCP Server 是提供能力的一端。它把某些工具、资源或数据源暴露给 AI 应用使用。

例如：

  • 文件系统 MCP Server
  • GitHub MCP Server
  • 数据库 MCP Server
  • 浏览器 MCP Server

# MCP Client

MCP Client 是连接 MCP Server 的客户端。通常由 AI 应用、编辑器、Agent 平台或桌面应用承担。

# MCP Host

MCP Host 是运行和管理 MCP Client 的宿主应用。

例如一个 AI 桌面应用可以作为 Host，负责连接多个 MCP Server，并把可用工具提供给模型。

# MCP Resource

MCP Resource 是 MCP Server 提供的资源，例如文件内容、数据库 schema、文档片段、项目配置等。

# MCP Tool

MCP Tool 是 MCP Server 暴露出来的可执行操作，例如查询数据库、读取文件、创建 issue、搜索文档等。

# MCP Prompt

MCP Prompt 是 MCP Server 提供的可复用提示模板，用于帮助模型完成特定任务。

# 9. RAG、Embedding 与向量数据库

# RAG

RAG 是 Retrieval-Augmented Generation 的缩写，中文常称为“检索增强生成”。

它的核心流程是：

  1. 用户提问
  2. 系统从知识库中检索相关资料
  3. 把资料连同问题一起交给模型
  4. 模型基于资料生成回答

RAG 常用于：

  • 企业知识库问答
  • 文档助手
  • 客服机器人
  • 法律、医疗、金融资料查询
  • 私有数据问答

# Retrieval

Retrieval 是“检索”。它指从数据库、搜索引擎、向量库或文件系统中找到与问题相关的信息。

# Embedding

Embedding，中文常称为“嵌入”或“向量表示”。它是把文本、图片、音频等内容转换成一串数字向量。

这些向量可以表示内容之间的语义相似度。

例如：

  • “苹果手机”
  • “iPhone”
  • “Apple mobile phone”

它们文字不同，但语义接近，向量距离也可能较近。

# Vector

Vector 是“向量”。在 AI 中，它通常是一组数字，用来表示文本、图片、音频等内容的特征。

# Vector Database

Vector Database，中文是“向量数据库”。它专门用于存储和检索向量数据。

常见用途：

  • 语义搜索
  • 文档问答
  • 推荐系统
  • 相似图片搜索
  • RAG 知识库

# Similarity Search

Similarity Search 是“相似度搜索”。它根据向量之间的距离或相似度，找出最接近的内容。

# Chunk

Chunk 是“文本块”或“分片”。在 RAG 中，长文档通常会被切分成多个 chunk，再分别进行 embedding 和检索。

切分过大可能导致检索不精准；切分过小可能丢失上下文。

# Reranking

Reranking 是“重排序”。在初步检索出一批候选内容后，系统会再次评估这些内容与问题的相关性，并重新排序。

# 10. 训练、微调与推理

# Training

Training 是“训练”。它指使用大量数据调整模型参数，使模型学习某种能力。

# Pretraining

Pretraining 是“预训练”。大模型通常先在海量通用数据上训练，学习语言、知识和推理基础。

# Fine-tuning

Fine-tuning 是“微调”。它是在已有模型基础上，用特定数据进一步训练，使模型更适合某个场景。

例如：

  • 客服话术
  • 法律文书
  • 医疗问答
  • 企业内部风格
  • 特定代码规范

# Supervised Fine-tuning

Supervised Fine-tuning，简称 SFT，中文是“监督微调”。它使用人工标注的输入输出样本来训练模型。

# RLHF

RLHF 是 Reinforcement Learning from Human Feedback 的缩写，中文常称为“基于人类反馈的强化学习”。

它通过人类偏好数据帮助模型生成更符合人类期望的回答。

# Inference

Inference，中文是“推理”或“推断”。它指模型已经训练好之后，接收输入并生成输出的过程。

用户日常调用 AI 写作、问答、生成代码，大多属于 inference。

# Parameters

Parameters 是“参数”。在模型中，它们是经过训练得到的数值，用来决定模型如何处理输入和生成输出。

模型参数越多，通常表示容量越大，但并不一定代表在所有任务上都更好。

# Weights

Weights 是“权重”。它是模型参数的一种说法，指神经网络中用于计算的数值。

# 11. 评估与质量

# Evaluation

Evaluation，简称 Eval，中文是“评估”。它用于测试模型或 AI 系统在某些任务上的表现。

评估指标可能包括：

  • 准确率
  • 召回率
  • 幻觉率
  • 响应速度
  • 成本
  • 用户满意度
  • 安全性

# Benchmark

Benchmark 是“基准测试”。它是一组标准化任务，用于比较不同模型或系统的能力。

# Hallucination

Hallucination，中文常称为“幻觉”。它指模型生成看似合理但实际错误、无来源或捏造的内容。

减少幻觉的方法：

  • 使用 RAG 提供可靠资料
  • 要求模型引用来源
  • 限制模型只基于给定材料回答
  • 对关键结果做人工审核
  • 使用工具验证事实

# Grounding

Grounding，中文可称为“依据化”或“落地到事实”。它指让模型回答建立在明确数据、文档、工具结果或真实上下文上。

# Guardrail

Guardrail，中文是“护栏”。它是限制模型行为、降低风险的一套规则或系统。

例如：

  • 拒绝危险请求
  • 过滤敏感信息
  • 限制输出格式
  • 防止泄露隐私
  • 对高风险领域增加人工审核

# 12. 安全、隐私与权限

# Secret

Secret 是“机密信息”。它包括 API Key、数据库密码、访问令牌、私钥等。

Secret 不应出现在：

  • 公共代码仓库
  • 日志
  • 截图
  • 前端代码
  • 明文配置文件

# Environment Variable

Environment Variable，中文是“环境变量”。它常用于保存配置和密钥，让程序运行时读取，而不是把敏感信息写死在代码中。

例如：

OPENAI_API_KEY=your_api_key_here

# Access Token

Access Token 是“访问令牌”。它用于证明某个用户、应用或服务有权限访问资源。

# Authentication

Authentication，中文是“身份认证”。它回答的问题是：“你是谁？”

例如登录账号、验证密码、使用 API Key。

# Authorization

Authorization，中文是“授权”。它回答的问题是：“你能做什么？”

例如某个用户可以读取文件，但不能删除文件。

# Data Privacy

Data Privacy，中文是“数据隐私”。它关注用户数据如何被收集、存储、使用、共享和删除。

AI 应用尤其需要注意：

  • 用户输入是否包含敏感信息
  • 数据是否会用于训练
  • 日志是否保存明文
  • 是否符合企业或地区合规要求

# PII

PII 是 Personally Identifiable Information 的缩写，中文是“个人身份识别信息”。

例如：

  • 姓名
  • 手机号
  • 身份证号
  • 邮箱
  • 地址
  • 银行卡信息
  • 面部照片

# 13. 模型部署与工程

# Deployment

Deployment，中文是“部署”。它指把模型或 AI 应用发布到可使用的环境中。

# Latency

Latency 是“延迟”。它表示从用户发出请求到收到响应之间的时间。

AI 应用中常见延迟来源：

  • 模型推理时间
  • 网络传输
  • 工具调用
  • 数据库检索
  • 长文本生成

# Throughput

Throughput 是“吞吐量”。它表示系统在单位时间内能处理多少请求或 token。

# Concurrency

Concurrency 是“并发”。它表示系统同时处理多个请求的能力。

# Caching

Caching，中文是“缓存”。它把常用结果临时保存起来，以减少重复计算、降低成本、提高速度。

# Observability

Observability，中文可称为“可观测性”。它用于监控系统运行状态，帮助发现错误、性能问题和成本异常。

常见手段：

  • 日志
  • 指标
  • 链路追踪
  • 错误告警
  • token 用量统计

# Logging

Logging 是“日志记录”。AI 应用通常会记录请求、响应、错误、耗时、token 用量等。

注意：日志中不要保存敏感信息或完整隐私数据。

# 16. 实用建议

# 使用 AI API 时

# 写 Prompt 时

  • 给足背景
  • 说明输出格式
  • 提供好坏示例
  • 对复杂任务分步骤

  • 合理切分 chunk
  • 使用 embedding 做语义检索
  • 对检索结果进行 reranking
  • 要求模型基于来源回答
  • 对答案提供引用或出处

# 构建 Agent 时

  • 限制工具权限
  • 记录关键操作
  • 对写入、删除、支付、发消息等操作加确认
  • 给 Agent 明确目标和边界
  • 设计失败恢复机制

# 双指针与滑动窗口

# 题目描述

给定一个整数数组 nums 和一个整数 k，请你返回数组中不同的 k-diff 数对的数目。

k-diff 数对定义为一个整数对 (nums[i], nums[j])，需满足以下全部条件：

  1. 0 <= i, j < nums.length

  2. i != j

  3. |nums[i] - nums[j]| == k

双指针是一个解决暴力枚举的技巧，实则应该连算法都算不上，双指针的一般使用前提是数组内部自然有序，也可以后期自主排列

那么就本题而言，假设存在一个数组

int arr={1,2,3,4,5}
//假设我要使得两个数字的相减的差为3

# set

**集合（set）**是一个内部自动有序且不含重复元素的容器，它可以在需要删除重复元素的情况下大放异彩，节省时间，减少思维量。

set 就是关键字的简单集合，当只是想知道一个值是否存在时，set 是最有用的。set 内部采用的是一种非常高效的平衡检索二叉树 ：红黑树（Red-Black Tree），也称为 RB 树，RB 树的统计性能要好于一般平衡二叉树。在 set 中每个元素的值都唯一，而且系统能根据元素的值自动进行排序，set 中元素的值不能直接被改变。

标准库提供 set 关联容器分为：

按关键字有序保存元素：set（关键字即值，即只保存关键字的容器）、multiset（关键字可重复出现的 set）；
无序集合：unordered_set（用哈希函数组织的 set）、unordered_multiset（用哈希函数组织的 set，关键字可以重复出现）

# Vector

向量（Vector）是一个封装了动态大小数组的顺序容器（Sequence Container）,跟任意其它类型容器一样，它能够存放各种类型的对象。可以简单的认为，向量是一个能够存放任意类型的动态数组

Vector的数据安排以及操作方式，与array(数组)非常相似，两者的唯一差别在于空间的运用的灵活性。

  • Array是静态空间，一旦配置了就不能改变，要换大一点或者小一点的空间，可以，一切琐碎得由自己来，首先配置一块新的空间，然后将旧空间的数据搬往新空间，再释放原来的空间。
  • Vector是动态空间，随着元素的加入，它的内部机制会自动扩充空间以容纳新元素。因此vector的运用对于内存的合理利用与运用的灵活性有很大的帮助，我们再也不必害怕空间不足而一开始就要求一个大块头的array（数组）了
  • Vector的实现技术，关键在于其对大小的控制以及重新配置时的数据移动效率，一旦vector旧空间满了，如果客户每新增一个元素，vector内部只是扩充一个元素的空间，实为不智，因为所谓的扩充空间(不论多大)，一如刚所说，是”配置新空间-数据移动-释放旧空间”的大工程,时间成本很高，应该加入某种未雨绸缪的考虑，稍后我们便可以看到vector的空间配置策略

# hash_map原理简介
这是一节让你深入理解hash_map的介绍，如果你只是想囫囵吞枣，不想理解其原理，你倒是可以略过这一节，但我还是建议你看看，多了解一些没有坏处。

hash_map基于hash table（哈希表）。 哈希表最大的优点，就是把数据的存储和查找消耗的时间大大降低，几乎可以看成是常数时间；而代价仅仅是消耗比较多的内存。然而在当前可利用内存越来越多的情况下，用空间换时间的做法是值得的。另外，编码比较容易也是它的特点之一。

其基本原理是：使用一个下标范围比较大的数组来存储元素。可以设计一个函数（哈希函数，也叫做散列函数），使得每个元素的关键字都与一个函数值（即数组下标，hash值）相对应，于是用这个数组单元来存储这个元素；也可以简单的理解为，按照关键字为每一个元素“分类”，然后将这个元素存储在相应“类”所对应的地方，称为桶。

但是，不能够保证每个元素的关键字与函数值是一一对应的，因此极有可能出现对于不同的元素，却计算出了相同的函数值，这样就产生了“冲突”，换句话说，就是把不同的元素分在了相同的“类”之中。 总的来说，“直接定址”与“解决冲突”是哈希表的两大特点。

hash_map，首先分配一大片内存，形成许多桶。是利用hash函数，对key进行映射到不同区域（桶）进行保存。其插入过程是：

得到key
通过hash函数得到hash值
得到桶号(一般都为hash值对桶数求模)
存放key和value在桶内。
其取值过程是:

得到key
通过hash函数得到hash值
得到桶号(一般都为hash值对桶数求模)
比较桶的内部元素是否与key相等，若都不相等，则没有找到。
取出相等的记录的value。
hash_map中直接地址用hash函数生成，解决冲突，用比较函数解决。这里可以看出，如果每个桶内部只有一个元素，那么查找的时候只有一次比较。当许多桶内没有值时，许多查询就会更快了(指查不到的时候).

由此可见，要实现哈希表, 和用户相关的是：**hash函数**和**比较函数**。这两个参数刚好是我们在使用hash_map时需要指定的参数。

# 什么时候需要用hash_map，什么时候需要用map

总体来说，hash_map 查找速度会比map快，而且查找速度基本和数据数据量大小，属于常数级别;而map的查找速度是log (n)级别。并不一定常数就比log(n)小，hash还有hash函数的耗时，明白了吧，如果你考虑效率，特别是在元素达到一定数量级时，考虑考虑hash_map。但若你对内存使用特别严格，希望程序尽可能少消耗内存，那么一定要小心，hash_map可能会让你陷入尴尬，特别是当你的hash_map对象特别多时，你就更无法控制了，而且hash_map的构造速度较慢。
