# iLoboke足球机器人比赛先进算法报告

检索日期：2026-05-18
资料范围：本文只使用网上公开资料，包括比赛规则文件、赛事报道、SOM/iLoboke相关文章、RoboCup Small Size League 公开资料、Rocos文档以及公开论文。报告中对 getballopp 的解释属于基于任务命名和比赛语境的工程推断，未把它写成官方公开定义。

# 摘要

iLoboke足球机器人比赛本质上是一个小型多智能体足球系统：机器人依赖上方视觉系统获得球、己方机器人和对方机器人的位置；策略端依据比赛状态运行脚本；技能和规划层把“拿球、传球、射门、接球、拦截、防守”等任务转换成速度、朝向、踢球和吸球等底层动作。第二十七届中国机器人及人工智能大赛规则明确指出，创非凡iLoboke足球项目重点考察图像处理、自动决策、路径规划、自主避障、C++/Lua混合程序设计，以及多智能体协同和动态环境控制能力1。

从比赛任务看，省赛常见目标是两个进攻机器人在五个固定点位完成定位球战术，除点球外射门前至少传接一次，且要求在15秒内完成1。国赛则升级到3V3对抗，要求在原有配合基础上增加对抗、防守等策略1。因此，先进算法不应只理解为某个单独函数，而应理解为“感知、状态估计、任务切换、角色分配、路径规划、传射决策、防守压迫、实时调参”的组合系统。

# 一、比赛平台与系统基础

# 1.1 iLoboke与SOM平台

iLoboke挑战赛规则文件说明，参赛学校使用 iLoboke足球机器人专用软件 SOM3.4.2 编写程序并在现场运行，平台提供虚拟仿真环境，因此无实物也可进行调试1。较早的iLoboke挑战赛方案也说明，SOM软件是与北京启创远景科技有限公司iLoboke足球机器人配套的操作管理软件，为用户提供人机交互和二次开发平台2。

在软件结构上，SOM后端主要实现决策调度，并分为策略脚本层、技能规划层和基础SDK层；其中策略脚本层使用 Lua，技能规划层和基础SDK层使用 C++2。这说明 getball、passball 一类任务通常不是孤立动作，而是 Lua 层组织比赛状态、C++ 层执行运动和踢球技能的协作结果。

# 1.2 典型硬件与赛场约束

第二十七届规则规定，iLoboke赛场有效尺寸为6m×4m，场地上方约3.2m处架设高速相机采集图像，比赛用球为橘黄色高尔夫球1。省赛中通常由2台进攻机器人完成5个固定点位的定位球进攻，防守侧包含守门员和场景放置的防守机器人；决赛阶段则为双方各3台机器人对抗1。

这些规则直接决定算法目标：省赛强调稳定、快速、少犯规的定位球战术；国赛强调对抗条件下的持球、传球、拦截、压迫和防守转换。赛事报道也显示，优秀队伍会围绕“传球速度优化”“精准识别位置”“高速传递”等方面持续调试运动算法3。

# 二、整体算法架构

小型足球机器人系统通常可拆为四个闭环：视觉感知、策略决策、通信下发、机器人执行。Rocos文档对小型足球机器人系统的描述与iLoboke场景高度相似：外部全局图像系统完成识别，核心决策系统部署策略与规划，通信模块下发指令，机器人硬件执行运动、踢球、吸球盘带等动作4。RoboCup SSL官方技术概览也说明，SSL-Vision负责把场上相机图像转换为机器人和球的坐标，再通过网络广播给队伍；同时原始视觉检测会有噪声、丢失和无速度信息，因此还需要进一步处理或跟踪5。

flowchart LR
    A["高速相机 / 公共视觉"] --> B["坐标识别与状态估计"]
    B --> C["策略脚本层 Lua"]
    C --> D["任务层 Task"]
    D --> E["技能层 Skill / C++"]
    E --> F["轨迹规划与避障"]
    F --> G["无线通信与底层控制"]
    G --> H["机器人运动、吸球、踢球"]

在策略层上，Rocos把核心策略拆为 Tactic、Play、Task、Skill、Plan 五层：Play 是多机器人配合策略，Task 是对泛化 Skill 的具体封装，Skill 是移动、射门、传球、拦截等上层控制，Plan 负责把目标点转化为速度和角速度6。虽然Rocos不是SOM官方手册，但它反映了小型足球机器人领域常用的软件分层方式，可用于解释iLoboke中“脚本状态、任务函数、C++技能、轨迹规划”之间的关系。

# 四、getball：从“冲向球”到“稳定控球”

# 4.1 任务定义

getball 的目标是让机器人从当前位姿到达合适的抓球位姿，并在不犯规、不把球顶飞、不被对手抢先触球的情况下获得控球权。基础写法可能只是让机器人朝球移动，但比赛中的先进 getball 应至少解决四个问题：从哪一侧接近球、以什么朝向接近球、何时启动吸球/盘带、失败后如何恢复。

# 4.2 抓球前置点算法

简单追球容易出现“从侧面或后方顶球”的问题。更稳的做法是根据后续动作确定目标方向，例如传球给队友、射门到球门或避开防守者，然后在球的反方向生成一个前置点：

目标方向 dir = normalize(target - ball)
抓球前置点 pre = ball - d_pre * dir
机器人目标朝向 theta = angle(target - robot)

其中 target 可以是接球队友、球门空档或下一状态需要的目标点。机器人先到 pre，再沿 dir 方向小速度贴近球。这样做的好处是，机器人头部、球和后续传射目标大致共线，拿到球后可以直接进入 passball 或 shoot。

# 4.3 状态切换与容错

Rocos文档说明，Play中的 switch 函数每帧执行，可依据距离、角度、速度、位置、是否持球、是否踢球、球出界、球消失、机器人数量变化、裁判盒信息和时间等条件切换状态9。在 getball 中，先进状态机不应只判断“离球距离小于阈值”，而应综合判断：

  1. 球是否位于机器人前方可吸球扇区。
  2. 机器人朝向与球-目标方向夹角是否足够小。
  3. 红外或控球状态是否表明球已被吸住。
  4. 机器人和球是否接近边界或禁区。
  5. 对手是否会更早到达球。

推荐状态机如下：

stateDiagram-v2
    [*] --> ApproachBehindBall
    ApproachBehindBall --> AlignToTarget: 到达抓球前置点
    AlignToTarget --> SlowCapture: 朝向误差小
    SlowCapture --> Controlled: 控球成功
    SlowCapture --> Recover: 球侧偏或顶球
    Recover --> ApproachBehindBall: 重算前置点
    Controlled --> PassBall: 传球窗口打开
    Controlled --> Shoot: 射门窗口打开

# 4.4 关键改进点

第一，getball 要和路径规划联动。禁区、边界、对方机器人和己方队友都应进入代价地图，避免为了拿球而进入禁区或碰撞防守机器人。第二，getball 要具备低速末端控制，接近球时降低速度并锁定朝向，减少把球推偏的风险。第三，getball 应对视觉噪声鲁棒：SSL官方资料指出视觉检测有噪声和丢失问题5，因此工程上应加入位置滤波、球速估计、短时丢球保持和异常重规划。

# 五、passball：从“把球踢给队友”到“动态传射决策”

# 5.1 规则驱动的传球价值

省赛定位球规则要求，除点球外，射门前至少完成一次传接配合1。这意味着 passball 不是可选动作，而是得分链路的一部分。公开赛事报道中也提到，参赛队会围绕传球速度优化、位置识别和高速传递进行调试3。

# 5.2 传球力度与球模型

ZJUNlict在RoboCup SSL 2019冠军论文中指出，传球力度过小会给对手足够时间拦截，力度过大则可能导致队友无法在有限时间内接球，因此需要基于球模型计算合适传球力度10。论文把传球时间拆为接球机器人到达时间和到达后的调整缓冲时间，并结合传球距离估计初速度10。

映射到iLoboke比赛中，passball 可以采用如下计算框架：

候选接球点 p
传球距离 d = |p - passer|
接球者到达时间 t_robot = time_to_reach(receiver, p)
缓冲时间 t_buffer = 0.2s ~ 0.4s
目标传球时间 t_pass = t_robot + t_buffer
传球速度 v = ball_model_inverse(d, t_pass)

在省赛中，两个进攻机器人通常预设“拿球者-接球者-射门者”的链路。先进做法不是固定传球点，而是在每个固定点位启动后，根据球、守门员、防守机器人和己方接球者的位置实时选择接球点。

# 5.3 传球点搜索与拦截风险

ZJUNlict论文提出的动态传射算法包括四个模块：传球点模块计算所有传球点可行性并选择最佳点，跑位点模块寻找进攻威胁更高的位置，决策模块判断何时传球或射门，技能模块保证动作执行准确10。该论文还以拦截时间、接球点射门角度、接球点到球门距离、射门折射角、己方先到与对方先到的时间差等特征评价传球策略10。

因此，passball 的先进评分函数可写为：

score(p) =
  w1 * pass_safety(p)
+ w2 * receiver_arrival_margin(p)
+ w3 * shooting_angle_after_receive(p)
+ w4 * distance_to_goal_value(p)
+ w5 * no_foul_margin(p)
- w6 * collision_risk(p)

其中 pass_safety 由“对手是否能在球到达前截到球”决定，receiver_arrival_margin 表示己方接球者比对手更早到达的时间差，no_foul_margin 用于避免进攻机器人进入禁区、二次触球或碰撞防守机器人等犯规。

# 5.4 接球同步

passball 不只是传球者动作，还包括接球者的 receiveball。Rocos的任务解析文档显示，一个Task可以返回执行函数和匹配位置，Lua层还可配置踢球模式、目标朝向、踢球精度、力度和特殊flag，从而把非运动指令与运动本身解耦11。这启发我们把传球设计为“双机器人同步任务”：

  1. 传球者根据接球者预计到达时间计算球速。
  2. 接球者提前移动到接球点，并朝向来球方向。
  3. 接球者在球到达前进入低速等待或微调状态。
  4. 传球者踢球后，接球者切换到吸球或一脚射门状态。

# 六、getballopp：对手控球场景下的抢球、封堵与压迫

# 6.1 资料边界

本次在线检索没有找到 getballopp 的官方公开定义。按常见命名习惯，它很可能表示 “get ball opponent” 或“对方控球/对方取球时的处理任务”。在报告中，getballopp 被定义为：当对手更接近球、对手准备接球、或对手即将形成射门/传球威胁时，己方机器人执行抢球、截球、封堵传球线或压迫持球人的综合任务。

# 6.2 对手触球时间预测

先进 getballopp 的第一步是判断“谁会先碰到球”。ZJUNlict论文中把控球率计算与双方拦截时间联系起来：若一方拦截时间更短，则球被认为由该方控制10。这给 getballopp 一个清晰指标：

t_our = min time_to_intercept(己方机器人, ball_trajectory)
t_opp = min time_to_intercept(对方机器人, ball_trajectory)
if t_opp + margin < t_our:
    进入 getballopp / 压迫防守
else:
    进入 getball / 主动拿球

若对手先到，直接冲球往往会造成碰撞或被轻松过掉，更好的动作是抢占对手下一步传球/射门路径。

# 6.3 三类策略

第一类：抢截球。 当球是滚动球或无人稳定控球时，己方机器人计算球轨迹与自身可达区域的交点，选择最早可达且不越界、不入禁区的拦截点。

第二类：封传球线。 当对手持球但射门角度不大时，机器人站到“对手持球点-潜在接球者”或“对手持球点-球门”的连线上，优先阻断高威胁传球路线。

第三类：压迫持球人。 当对手拿球且转身空间有限时，机器人从侧前方逼近，迫使对手向低威胁方向移动。这里不能直接撞击对方机器人，因为第二十七届规则把进攻机器人碰撞防守机器人列为犯规之一1；在3V3对抗中也需要用规则安全的方式防守。

# 6.4 getballopp 状态机

stateDiagram-v2
    [*] --> EstimatePossession
    EstimatePossession --> InterceptBall: 球未被稳定控制
    EstimatePossession --> BlockLane: 对手准备传球
    EstimatePossession --> PressHolder: 对手稳定持球
    InterceptBall --> ClearOrPass: 己方截到球
    BlockLane --> GetBall: 对手传球被破坏
    PressHolder --> RetreatDefend: 对手突破或己方失位
    ClearOrPass --> [*]

getballopp 的先进性不在于“更快冲向对手”，而在于用预测和几何控制缩小对手选择空间。它需要和 defend、goalie、passball 共享同一套威胁评分：对手离球门越近、射门角越大、传球线越通畅，则防守动作越激进；反之则保持阵型，避免无谓出击。

# 七、射门、定位球与无球跑位算法

# 7.1 射门决策

射门不是越早越好。ZJUNlict论文指出，随意射门可能导致丢失球权，条件过严又会降低进攻效率，因此需要结合射门角和拦截预测来判断是否射门10。iLoboke省赛中，射门前还要满足传接配合要求，且直接射门会被判犯规或进球无效1。

推荐射门判断：

can_shoot =
    has_required_pass
and shot_angle > angle_threshold
and opponent_intercept_time(ball_to_goal) > ball_arrival_time + margin
and robot_not_in_penalty_area
and shot_power_within_limit

# 7.2 定位球战术

省赛的五个点位包含后场区、开球点、前场区、角球区、点球点，具体点位赛前不公布1。这要求算法不能只写死坐标，而应根据开赛后裁判放球点实时生成战术。一个稳健的省赛流程是：

  1. 读取球点和障碍机器人位置。
  2. 由最近且角度最优的机器人执行 getball。
  3. 另一台机器人计算接球点，避开防守机器人和禁区。
  4. 持球者执行 passball。
  5. 接球者判断直接射门或微调后射门。
  6. 全程监控15秒倒计时，若超过安全时间则选择快速射门或弃守当前战术。

# 7.3 无球跑位

ZJUNlict论文把前场划分为动态区域，并根据球的位置改变区域；然后把区域栅格化，对每个点按接近球门、离球距离、面对球门角度、对手防守到达时间、避免陷入对手防守等标准评分10。在iLoboke中，这类算法可用于 GoRecePos、ReceiveBall 或自定义跑位任务：接球者不是站在固定点，而是实时寻找“更容易接球、更容易射门、更不容易被拦截”的点。

# 7.4 反盯防移动

ZJUNlict还提出 Drag skill：当机器人到达目的地后容易被对手盯防时，通过先诱导防守者加速，再反向加速拉开距离，以提升安全接球概率10。在iLoboke 3V3对抗中，类似思路可用于接球机器人：不要在接球点静止等待，而是在不越位、不入禁区、不撞人的前提下做小范围横移，使对手难以提前卡死传球线。

# 八、角色分配与多机器人协同

角色分配决定哪个机器人去拿球、哪个机器人接球、哪个机器人补防。Rocos文档说明，在多智能体场景中，匹配规则用于把角色任务分配给具体机器人；实时匹配可保证总代价最小，底层使用二分图匹配中的匈牙利算法12。这类机制对iLoboke也很重要，因为比赛中机器人可能因位置、电量、碰撞或识别异常导致最优执行者变化。

一个高级角色分配代价函数可以包括：

cost(robot, role) =
  a1 * 到任务目标距离
+ a2 * 当前朝向误差
+ a3 * 到达时间
+ a4 * 可能碰撞风险
+ a5 * 是否带球/是否刚踢球
+ a6 * 禁区和边界惩罚
+ a7 * 通信或硬件状态惩罚

在省赛中，角色可较稳定：一台主抓球/传球，一台主接球/射门，守门员由组委会或己方固定。国赛3V3中，角色应动态切换：前锋丢球后可立刻转为压迫者，中场可转为接应点，守门员在球远离危险区时可上提参与出球。

# 九、路径规划与避障

路径规划层把“去球后方”“去接球点”“封堵传球线”等目标转换为可执行轨迹。SOM规则要求场地有禁区、边界、球门和防守机器人，且进攻机器人进入禁区、二次触球、碰撞防守机器人都会影响成绩1。因此，路径规划不仅要快，还要规则安全。

推荐采用分层规划：

  1. 全局目标生成：由 getball、passball、getballopp 计算目标点。
  2. 规则过滤：删除禁区内、边界外、可能碰撞的目标点。
  3. 局部避障：对动态机器人设置膨胀半径，避免高速贴近。
  4. 速度规划：接近球、接近对手、接近边界时降速。
  5. 重规划触发：球速变化、视觉丢失、对手突然靠近、队友位置变化时重算路径。

对比赛成绩而言，路径规划的评价指标不是单纯最短路径，而是“最短时间内完成有效传射且不犯规”。这也是为什么优秀算法常把路径规划和传球决策合并考虑：某个传球点看似角度好，但如果接球者到达时间太长或路径穿过对手防守区，就不是好点。

# 十、针对不同赛段的策略建议

# 10.1 省赛定位球

省赛目标清晰：五个点位、15秒、至少一次传接配合、进球优先、用时次之1。建议采用“少状态、高确定性”的策略：

场景 首选策略 重点算法
后场区 稳定传球后远射或推进射门 长距离传球力度、接球者提前跑位
开球点 快速一传一射 固定开球模板、射门角选择
前场区 短传后低延迟射门 接球同步、守门员角度判断
角球区 斜传到门前空档 传球线避障、接球者横向跑位
点球点 直接射门或假动作射门 射门角、守门员位置预测

# 10.2 国赛3V3对抗

中国大学生在线报道提到，相关赛事决赛阶段需要自行携带机器人，并且国赛赛制升级为3V3对抗，每队派出3台机器人进行约5分钟比赛，并新增防守策略设计13。规则文件也说明，国赛3V3中双方共有六台机器人，参赛队需在原有配合基础上增加对抗和防守策略1。

3V3中建议使用“攻防状态机”：

flowchart TD
    A["开球 / 普通比赛"] --> B{"己方是否先到球"}
    B -->|是| C["getball 控球"]
    C --> D{"传球窗口是否安全"}
    D -->|是| E["passball + receiveball"]
    D -->|否| F{"射门窗口是否安全"}
    F -->|是| G["shoot"]
    F -->|否| H["off-ball running / 重新组织"]
    B -->|否| I["getballopp 压迫或截球"]
    I --> J{"对手威胁等级"}
    J -->|高| K["block lane / goalie 协防"]
    J -->|低| L["保持阵型 / 等待失误"]

# 十一、主要风险与改进方向

风险 现象 改进方向
视觉误差 球坐标跳动、机器人误判持球 滤波、短时预测、状态滞回
抓球不稳 机器人绕球、顶球、错过球 前置点、低速末端、朝向闭环
传球失败 球速过大或过小、队友没到位 球模型、到达时间预测、传球缓冲
被对手拦截 传球线被封堵 拦截时间评估、传球点搜索
犯规 入禁区、二次触球、碰撞 规则约束加入规划代价
状态切换抖动 反复在getball/passball之间切换 状态滞回、最短状态保持时间
国赛对抗不适应 只会定位球，不会防守 getballopp、盯防、守门、阵型切换

未来可以重点改进三方面。第一，用在线日志自动调参：记录每次传球距离、力度、接球成败、射门角度和用时，回放后优化权重。第二，引入对手建模：统计对手常用传球方向、守门员移动习惯、压迫强度，让 getballopp 更像“预判”而不是“追赶”。第三，提高技能层鲁棒性：针对抓球、接球、踢球分别做小规模测试集，使任务在仿真和实物之间表现更一致。

# 结论

iLoboke足球机器人比赛的先进算法不是单个函数的堆叠，而是围绕比赛目标形成的闭环系统。getball 的核心是稳定获得球权，关键在于前置点、朝向控制、吸球判断和异常恢复；passball 的核心是满足规则并创造高质量射门机会，关键在于球模型、传球点搜索、接球同步和拦截风险评估；getballopp 虽未找到官方公开定义，但从3V3对抗需求看，它应被设计为对手控球时的抢截、封堵和压迫任务，核心在于预测双方拦截时间、控制传球线和避免犯规。

如果要在比赛中取得稳定成绩，建议把算法优先级排为：先保证 getball 和 passball 的成功率，再优化射门决策和定位球模板，最后加入 getballopp、防守阵型和无球跑位。省赛靠稳定传射拿分，国赛靠攻防切换和对抗决策拉开差距。
